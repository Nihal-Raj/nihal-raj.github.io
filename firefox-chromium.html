Firefox and Chromium

Last edited: March 3, 2025

A browser’s sandbox isolates processes to limit the damage of an exploit. Browsers handle untrusted input by design, making sandboxing critical. Without it, a vulnerability could compromise the entire system; with it, an attacker needs a sandbox escape too. But not all sandboxes are equal—strength lies in granularity and restrictions.

1.1 Site Isolation

Chromium introduced site isolation in 2018, running each website in its own sandboxed renderer process. This isolates exploits and mitigates side-channel attacks like Spectre by leveraging process boundaries. As of 2025, Chromium’s implementation has matured further, with optimizations reducing performance overhead while maintaining strict separation.

Firefox’s Fission project, fully rolled out in Firefox 95 (late 2021), aimed to match this. By March 2022, Fission was immature, inheriting weaknesses from Firefox’s baseline sandbox. By 2025, Fission has improved, with better site separation in Firefox 123 and later. However, it still lags Chromium in maturity. Cross-site leaks—where a compromised process accesses another site’s data—have been patched incrementally, but Firefox’s content process sandbox remains less restrictive, undermining Fission’s gains.

1.2 Windows

On Windows, Chromium uses a robust sandbox with strict process mitigations, like Arbitrary Code Guard (ACG) and Code Integrity Guard (CIG), enabled by default. Firefox, as of 2022, lacked these, exposing it to basic code execution exploits. By 2025, Firefox has adopted some Windows mitigations—ACG was added in Firefox 115 (2023)—but CIG remains absent, and its sandbox still permits broader system access than Chromium’s.

1.3 Linux

1.3.1 Linux Sandbox Escapes

In 2022, Firefox’s Linux sandbox had trivial escapes, like exposing X11 and PulseAudio to the content process. Chromium sandboxed these via dedicated processes. By 2025, Firefox has tightened this: X11 access is restricted in Firefox 120 (2024), and PulseAudio is now proxied. However, Chromium’s sandbox remains stricter, with fewer escape vectors due to its granular process model.

1.3.2 seccomp-bpf

Linux’s seccomp-bpf filters syscalls to reduce kernel attack surface. In 2022, Firefox’s filter was lax, barely restricting ioctls—a massive attack surface. Chromium allowed only necessary ioctls. By 2025, Firefox’s seccomp filter (updated in Firefox 118, 2023) blocks more ioctls, but Chromium’s filter is still more restrictive, covering a broader range of syscalls and arguments.

1.4 Android

Firefox on Android adopted a multi-process model in 2021 but lacked sandboxing beyond the OS’s app sandbox in 2022. Chromium used isolatedProcess and a strict seccomp-bpf filter. In 2025, Firefox (version 125+) has basic sandboxing for content processes, but it’s less mature than Chromium’s, which now includes enhanced site isolation and tighter JNI restrictions.

1.5 Missing Processes

Chromium’s granular process model—separate GPU, network, and RDD processes—allows tight sandboxing. In 2022, Firefox merged these into parent or content processes, weakening restrictions. By 2025, Firefox has a GPU process on Linux (since Firefox 119, 2023), but it’s unsandboxed, and other processes remain consolidated, trailing Chromium’s design.

2. Exploit Mitigations

2.1 Arbitrary Code Guard and Code Integrity Guard

Arbitrary Code Guard (ACG) prevents the introduction of new executable code in a process, while Code Integrity Guard (CIG) ensures only signed binaries execute. These Windows-specific mitigations are critical for thwarting exploits. Below is a comparison of their application across key processes in Chromium and Firefox as of March 3, 2025, updated from the original analysis in March 2022.

Process Chromium ACG Firefox ACG Chromium CIG Firefox CIG
Browser (Main) No No Yes No
Renderer Yes (JITless) Yes Yes No
GPU No No Yes No
Network No No Yes No
Audio Yes No Yes No
Proxy Resolver Yes No Yes No
Icon Reader Yes No Yes No
Socket No Yes Yes Yes
RDD No No Yes Yes

Notes (March 3, 2025):

  • Chromium: ACG is enabled in renderer processes only with JITless mode due to JIT’s writable-executable memory needs; CIG remains broadly applied across all listed processes. No significant expansion since 2022 reflects an already robust baseline.
  • Firefox: ACG was added to the renderer process in Firefox 115 (2023) and is active in the now-enabled socket process (since Firefox 118, 2023). CIG is limited to socket and RDD processes, with no broader adoption. Firefox has improved but trails Chromium’s granularity.
  • Gap: Chromium’s wider ACG/CIG coverage across more processes underscores its stronger mitigation model, despite Firefox’s progress.

2.2 Control Flow Integrity (CFI)

CFI prevents code hijacking. Chromium uses Clang’s forward-edge CFI on Linux/Android and Control Flow Guard on Windows. Firefox lacked CFI in 2022. By 2025, Firefox 121 (2024) introduced coarse-grained forward-edge CFI on Linux, but it’s less comprehensive than Chromium’s, and backward-edge CFI (e.g., SafeStack) is still missing.

2.3 Untrusted Fonts Blocking

Chromium blocks untrusted fonts to reduce attack surface; Firefox didn’t in 2022. By 2025, Firefox 124 (2024) restricts font loading in sandboxed processes, catching up but not surpassing Chromium’s stricter defaults.

2.4 JIT Hardening

Both browsers use JIT compilers, requiring writable-executable memory—a security risk. Chromium’s JIT mitigations (e.g., partitioning) were stronger in 2022. In 2025, Firefox (via Firefox 122, 2024) has adopted similar hardening, narrowing the gap, though Chromium’s implementation remains more battle-tested.

2.5 Memory Allocator Hardening

Memory allocators manage heap memory, and hardening them reduces the risk of exploits like use-after-free or buffer overflows. In 2022, Chromium’s custom PartitionAlloc outclassed Firefox’s reliance on system allocators. As of March 3, 2025, the landscape has evolved:

  • Chromium: Uses PartitionAlloc (enhanced with MiraclePtr), featuring guard pages, randomization, and use-after-free mitigations like delayed freeing. It remains a hardened, exploit-resistant allocator.
  • Firefox: Adopted mozjemalloc with guard pages and randomization by Firefox 118 (2023), improving over system allocators. However, it lacks advanced features like MiraclePtr or delayed freeing.
  • Gap: Firefox has closed some ground, but Chromium’s allocator offers deeper protections, maintaining an edge against heap-based attacks.

2.6 Automatic Variable Initialisation

Uninitialized variables can leak data or enable control flow hijacking. In 2022, Chromium mitigated this with automatic stack variable initialization, while Firefox did not. As of March 3, 2025:

  • Chromium: Continues to use Clang’s -ftrivial-auto-var-init=zero in release builds, zero-initializing stack variables to prevent exploits.
  • Firefox: Has not enabled automatic initialization by default in release builds (e.g., Firefox 125), despite trials in Nightly (Bug 1495987). Performance trade-offs keep it opt-in, leaving gaps in protection.
  • Gap: Chromium’s systematic approach still outpaces Firefox, reducing a preventable class of vulnerabilities Firefox remains exposed to.

3. Miscellaneous

Firefox’s smaller market share doesn’t inherently make it safer—attackers target vulnerabilities, not popularity. Chromium’s larger user base drives faster patching and exploit mitigation development. Mozilla has improved Firefox since 2022, but its security focus still trails Chromium’s engineering rigor.
