Firefox and Chromium

Last edited: March 3, 2025

Note: This is the continuation to the original post by Madaidan, but since the original project hasn't updated in almost 3 years, I've decided to provide a new update on it with the help of Grok AI.


Firefox is often touted as a more secure browser due to Mozilla’s privacy-focused reputation. However, this perception conflates privacy with security—an error this article aims to correct. Here, we analyze Firefox’s security model against Chromium’s, focusing on their resistance to exploitation rather than privacy policies. Since the last update on March 19, 2022, both browsers have evolved, but significant gaps remain. This article covers process architecture, sandboxing, exploit mitigations, and more, updated to reflect the current state as of March 3, 2025.

1. Sandboxing

Sandboxing isolates applications and restricts their resource access, preventing vulnerabilities in one program from compromising the entire system. Modern browsers use sandboxing extensively: they run multiple processes (content, GPU, RDD, etc.) each with minimal privileges, processing untrusted input while minimizing attack surface. Without a sandbox, a browser exploit could hijack the entire system. With one, attackers need a second vulnerability to escape the sandbox, raising the bar significantly.

Still, sandboxes vary in quality. A poorly implemented sandbox provides little real protection. For instance, Firefox’s sandboxing has several known weaknesses, only some of which are listed below.

1.1 Site Isolation

Site isolation, introduced in Chromium in 2018, revamped its multi-process architecture by assigning each website its own sandboxed renderer process.

This prevents renderer exploits from accessing data from other websites and is essential for mitigating side-channel attacks like Spectre. OS-level protections only isolate at the process boundary, making process separation the most effective defense (Chromium Security Docs, USENIX Presentation, Chromium Side-Channel Threat Model). Current mitigations, like reducing JavaScript timer accuracy, are insufficient and fail to address the root issue (Chrome Developer Blog).

Firefox introduced Fission in Firefox 95 (Firefox Release Notes), but it remains less mature than Chromium’s site isolation and will require years to reach parity. Fission inherits the security weaknesses of Firefox’s content process sandbox (Bugzilla: 1505832, Bugzilla: 1484019) and is not a comprehensive solution.

Additionally, cross-site leaks in Fission allow compromised processes to bypass site isolation and access data from other sites (Bugzilla: 1707955).

1.2 Windows

On Windows, Chromium uses a robust sandbox with strict process mitigations, like Arbitrary Code Guard (ACG) and Code Integrity Guard (CIG), enabled by default. Firefox, as of 2022, lacked these, exposing it to basic code execution exploits. By 2025, Firefox has adopted some Windows mitigations—ACG was added in Firefox 115 (2023)—but CIG remains absent, and its sandbox still permits broader system access than Chromium’s.

As of Firefox 100, released in May 2022, Mozilla has implemented Win32k Lockdown for content processes on Windows. This security feature restricts access to certain system calls, reducing the attack surface and making sandbox escapes more difficult.

While Chromium implemented Win32k Lockdown earlier, in 2016, Firefox has since adopted similar measures to enhance its sandboxing capabilities on Windows.

1.3 Linux

Firefox's sandboxing on platforms like Linux is significantly weaker. The restrictions are relatively lenient, leaving it vulnerable to various sandbox escape exploits that have persisted for years. Furthermore, it exposes a substantial attack surface even within the sandbox environment.

On the other hand, Firefox no longer exposes X11 to the content process and proxies PulseAudio. Chromium restricts access to a dedicated audio service, addressing this issue.

Seccomp-BPF is a Linux sandboxing technology that enables the restriction of system calls available to a process, significantly reducing the kernel's attack surface and serving as a fundamental component of most Linux sandboxing mechanisms.

Firefox is not utilizing it the same way Chromium is (source, Bugzilla).

This is the main way out of the sandbox and is leveraged in most real world browser exploits. It's easier to exploit the kernel than the browser broker process in practice.

For X11 on Linux, Firefox does not have a separate GPU process, and therefore, no GPU process sandboxing is implemented.

Firefox lacks a separate audio process, unlike Chromium which has a dedicated audio service. In Firefox, audio functionality is integrated directly into the content process, leading to vulnerabilities such as the PulseAudio sandbox escape on Linux systems.

1.4 Android

Avoid using Firefox-based browsers like Firefox on Android due to:

Increased Vulnerability: Firefox lacks internal sandboxing, unlike Chromium which uses Android's isolatedProcess for strong isolation.

Sandboxing Deficiency: Even on desktop, Firefox's sandbox is weaker, particularly on Linux, with limited site isolation compared to Chromium. Android version lags further in sandbox improvements.

2. Exploit Mitigations

2.1 Arbitrary Code Guard and Code Integrity Guard

Arbitrary Code Guard (ACG) prevents the introduction of new executable code in a process, while Code Integrity Guard (CIG) ensures only signed binaries execute. These Windows-specific mitigations are critical for thwarting exploits. Below is a comparison of their application across key processes in Chromium and Firefox as of March 3, 2025, updated from the original analysis in March 2022.

Process Chromium ACG Firefox ACG Chromium CIG Firefox CIG
Browser (Main) No No Yes No
Renderer Yes (JITless) Yes Yes No
GPU No No Yes No
Network No No Yes No
Audio Yes No Yes No
Proxy Resolver Yes No Yes No
Icon Reader Yes No Yes No
Socket No Yes Yes Yes
RDD No No Yes Yes

Notes (March 3, 2025):

  • Chromium: ACG is enabled in renderer processes only with JITless mode due to JIT’s writable-executable memory needs; CIG remains broadly applied across all listed processes. No significant expansion since 2022 reflects an already robust baseline.
  • Firefox: ACG was added to the renderer process in Firefox 115 (2023) and is active in the now-enabled socket process (since Firefox 118, 2023). CIG is limited to socket and RDD processes, with no broader adoption. Firefox has improved but trails Chromium’s granularity.
  • Gap: Chromium’s wider ACG/CIG coverage across more processes underscores its stronger mitigation model, despite Firefox’s progress.

2.2 Control Flow Integrity (CFI)

CFI prevents code hijacking. Chromium uses Clang’s forward-edge CFI on Linux/Android and Control Flow Guard on Windows. Firefox lacked CFI in 2022. By 2025, Firefox 121 (2024) introduced coarse-grained forward-edge CFI on Linux, but it’s less comprehensive than Chromium’s, and backward-edge CFI (e.g., SafeStack) is still missing.

2.3 Untrusted Fonts Blocking

Chromium blocks untrusted fonts to reduce attack surface; Firefox didn’t in 2022. By 2025, Firefox 124 (2024) restricts font loading in sandboxed processes, catching up but not surpassing Chromium’s stricter defaults.

2.4 JIT Hardening

Both browsers use JIT compilers, requiring writable-executable memory—a security risk. Chromium’s JIT mitigations (e.g., partitioning) were stronger in 2022. In 2025, Firefox (via Firefox 122, 2024) has adopted similar hardening, narrowing the gap, though Chromium’s implementation remains more battle-tested.

2.5 Memory Allocator Hardening

Memory allocators manage heap memory, and hardening them reduces the risk of exploits like use-after-free or buffer overflows. In 2022, Chromium’s custom PartitionAlloc outclassed Firefox’s reliance on system allocators. As of March 3, 2025, the landscape has evolved:

  • Chromium: Uses PartitionAlloc (enhanced with MiraclePtr), featuring guard pages, randomization, and use-after-free mitigations like delayed freeing. It remains a hardened, exploit-resistant allocator.
  • Firefox: Adopted mozjemalloc with guard pages and randomization by Firefox 118 (2023), improving over system allocators. However, it lacks advanced features like MiraclePtr or delayed freeing.
  • Gap: Firefox has closed some ground, but Chromium’s allocator offers deeper protections, maintaining an edge against heap-based attacks.

2.6 Automatic Variable Initialisation

Uninitialized variables can leak data or enable control flow hijacking. In 2022, Chromium mitigated this with automatic stack variable initialization, while Firefox did not. As of March 3, 2025:

  • Chromium: Continues to use Clang’s -ftrivial-auto-var-init=zero in release builds, zero-initializing stack variables to prevent exploits.
  • Firefox: Has not enabled automatic initialization by default in release builds (e.g., Firefox 125), despite trials in Nightly (Bug 1495987). Performance trade-offs keep it opt-in, leaving gaps in protection.
  • Gap: Chromium’s systematic approach still outpaces Firefox, reducing a preventable class of vulnerabilities Firefox remains exposed to.

Oilpan (garbage collection for C++ objects) and MiraclePtr (use-after-free protection for non-Oilpan objects) are massive defenses against the main forms of memory corruption bugs in browsers (use-after-free). PartitionAlloc is also a major upgrade over jemalloc in Firefox.

For jemalloc to approach the security characteristics of Oilpan + MiraclePtr, it would need to transform from a conventional allocator into a full-fledged memory safety runtime. This means embedding garbage collection or reference tracking mechanisms, implementing pointer validation infrastructure, quarantining freed memory, integrating closely with compilers and language runtimes, and accepting significant performance and complexity overheads.

3. Miscellaneous

Firefox’s smaller market share doesn’t inherently make it safer—attackers target vulnerabilities, not popularity. Chromium’s larger user base drives faster patching and exploit mitigation development. Mozilla has improved Firefox since 2022, but its security focus still trails Chromium’s engineering rigor.
